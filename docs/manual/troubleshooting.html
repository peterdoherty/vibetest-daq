Troubleshooting

Solutions to common problems you may encounter when setting up or running Vibetest DAQ.

Installation Problems

"nidaqmx not available" error on start

This means the Python nidaqmx package cannot find the NI-DAQmx C driver. The driver must be installed separately from the Python package.

  1. Install the NI-DAQmx driver from NI Package Manager or ni.com.
  2. Reboot after installation.
  3. Verify that NI MAX can see your hardware.
  4. Then run pip install -e . again to ensure nidaqmx can link to the driver.

PySide6 / Qt import errors

If the GUI fails to start with an import error related to PySide6 or Qt, check that:

  • You are using Python 3.10 or later.
  • PySide6 is installed: pip install PySide6
  • On Linux, the system may need additional Qt runtime libraries (e.g., libxcb, libgl1).

Hardware Connection Issues

Device name mismatch

If the acquisition starts but immediately reports an error about an unknown device, the device names in the application do not match what NI MAX shows.

  1. Open NI MAX and expand Devices and Interfaces.
  2. Note the exact device names (e.g., cDAQ1Mod1, cDAQ1Mod2).
  3. In the GUI, enter these names in the Module 1 device and Module 2 device fields on the Acquire tab.
  4. For the CLI, edit the MODULE_1_DEVICE and MODULE_2_DEVICE constants in daq.py.
Tip: If you have multiple cDAQ chassis connected, the device names may be numbered differently than expected (e.g., cDAQ2Mod1 instead of cDAQ1Mod1). Always check NI MAX.

USB disconnection during recording

If the USB cable is disconnected during an acquisition, the current block read will fail with a DAQ error. The GUI will show the error in a dialog and stop recording. All previously written CSV files are intact. Reconnect the chassis and start a new acquisition.

Sample Rate Issues

Actual rate differs from requested rate

The NI 9230 derives its sample clock from internal dividers, so it can only produce certain exact rates. When you request a rate that is not achievable, the hardware silently rounds to the nearest supported value.

The application handles this correctly:

  • The GUI shows the actual rate in the Status section and status bar.
  • The CSV header records the actual rate, not the requested rate.
  • Block timestamps are computed using the actual rate.

If the offset is large enough to affect your analysis, either choose a rate that the hardware supports exactly, or account for the actual rate in your post-processing.

Why this matters: If downstream spectral analysis uses the requested rate instead of the actual rate from the CSV header, all frequency values will be off by a proportional amount. Always read the sample rate from the file header.

Data Quality Issues

Level meters always show zero

If the level meters stay at zero during recording:

  • Check that the accelerometers are connected to the correct module inputs.
  • Verify that IEPE excitation is set to 0.004 A (not 0). A value of 0 disables the constant-current source, and the sensors will not produce a signal.
  • Tap gently on the sensor to confirm it responds.

Level meters are red / signal looks saturated

Red meters or flat-topped waveforms indicate the signal is clipping. Possible causes:

  • The full-scale meter range is set too low — increase it to see if the actual peak is reasonable.
  • The vibration level genuinely exceeds the sensor/module range. You may need a less sensitive accelerometer for high-g measurements.
  • The sensitivity is set incorrectly. If the sensitivity value is too low, the computed g values will be artificially high. Double-check the sensor's datasheet.

Unexpected data values or noisy signal

  • Make sure the sensitivity setting matches your sensor. The wrong value will scale all readings by the wrong factor.
  • Check cable connections for loose BNC or LEMO connectors.
  • Verify that the sensor is securely mounted. A loose sensor will rattle and produce spurious high-frequency noise.

File and Disk Issues

Output directory errors

If the application cannot write to the output directory:

  • Make sure the path is valid and you have write permissions.
  • On network drives, check that the connection is stable — intermittent network drops will cause write failures.
  • Prefer a local disk for the output directory to minimize latency. Slow writes can cause the hardware buffer to overflow.

Running out of disk space

At 5 kHz with 6 channels, the application writes roughly 30 MB per minute. A one-hour recording produces about 1.8 GB. Monitor your available disk space for long acquisitions. The application will report a write error if the disk is full, but previously saved files are not affected.

GUI-Specific Issues

Settings not restoring on launch

Settings are stored using Qt's QSettings system. On Windows, this uses the registry. On macOS, it uses the preferences system. If settings are not restoring:

  • Check that the application was closed normally (not killed via Task Manager). Settings are saved on close and on start, so a hard kill before the first acquisition may not save the latest changes.
  • Try resetting by clearing the Qt settings. On Windows, look for HKEY_CURRENT_USER\Software\vibetest\daq in the registry editor.

"No channels selected" warning

You tried to start an acquisition with all channels disabled. Go to the Channels tab and enable at least one channel.

HDF5-Specific Issues

h5py import error

If you see an error about h5py not being found when selecting HDF5 format, reinstall the package:

pip install -e .

This will install h5py along with the other dependencies. On some systems, h5py requires the HDF5 C library; pip installs a bundled version automatically on most platforms.

Cannot open HDF5 files

HDF5 files are binary and cannot be viewed in a text editor. To inspect them:

  • HDFView — a free GUI viewer from The HDF Group.
  • h5dump — a command-line tool included with the HDF5 library (h5dump -A file.h5 shows attributes, h5dump -d data file.h5 shows dataset contents).
  • Python — a few lines with h5py (see Output Formats for examples).

Getting Help

If you encounter an issue not covered here:

  1. Check the terminal or console output for error messages — they often include specific NI-DAQmx error codes that you can look up in the NI documentation.
  2. Verify your hardware setup in NI MAX by running a test acquisition using NI's built-in tools.
  3. Review the Output Formats page to make sure you are parsing the output files correctly (CSV and HDF5 have different loading methods).
